## Findings

### RW35-001 — W3 backup behavior is not fault-characterized

- Lane/evidence: W3, \`P0.W3.BACKUP_BEHAVIOR\`
- Severity: high
- Affected decisions: ADR-32; Phase 0 task 23
- Evidence:
  - \`docs/research/hosted-web/phase-0/state-writers/backup-behavior.json:18\` presents a
    six-row fault matrix.
  - \`docs/research/hosted-web/phase-0/state-writers/backup-behavior.json:30\` labels the proof
    \`source-characterized\`.
  - \`docs/research/hosted-web/phase-0/state-writers/backup-behavior.json:31\` explicitly says
    faults were not injected into \`TeamBackupService\`.
  - \`test/architecture/hosted-web/phase-0/state-writers/state-writers.test.mjs:19\` through
    \`:125\` validates catalogs and the isolated SQLite/external-writer spikes, but never constructs or
    faults \`TeamBackupService\`.
  - The master plan at \`docs/hosted-web-e2e-completion-plan.md:4990\` requires the current async/sync
    enumeration, config-readiness, error-swallowing, prune, identity, and partial/mtime restore paths to
    be characterized against a fault-injected fixture.
- Why this blocks adoption: source reading supports the unsafe \`legacy_unverified\` conclusion, but it
  does not prove the exact observable fault outcomes required by the packet. Avoiding production edits
  does not prevent a marker-owned test from exercising existing production code.
- Required correction: add a temporary marker-owned \`TeamBackupService\` fixture covering the async and
  shutdown copy paths, unreadable/copy failures, corrupt/missing config restore, registry rebuild,
  manifest/publication failure and prune behavior. Record exact commands/results and retain the
  \`legacy_unverified\` classification unless the fixture contradicts it.

### RW35-002 — W5 does not schedule several claimed boundaries

- Lane/evidence: W5, \`P0.W5.SNAPSHOT_HANDOFF_SCHEDULER\` and
  \`P0.W5.EFFECT_RECOVERY_MATRIX\`
- Severity: critical
- Affected decisions: ADR-33, ADR-34
- Evidence:
  - \`scripts/hosted-web/phase-0/recovery-events/model.mjs:11\` defines snapshot mutation pauses without
    \`before_commit\` or \`after_commit\`.
  - \`scripts/hosted-web/phase-0/recovery-events/model.mjs:32\` merely appends both commit pause labels
    inside one indivisible \`commitMutation()\`.
  - \`scripts/hosted-web/phase-0/recovery-events/model.mjs:127\` generates schedules only from the ten
    non-commit pause names, while the report later lists commit labels as exercised.
  - \`scripts/hosted-web/phase-0/recovery-events/model.mjs:275\` appends compensation and event-publication
    pause labels after recovery without crashing or resuming at those boundaries.
  - \`test/architecture/hosted-web/phase-0/recovery-events/recovery-events.test.mjs:57\` and \`:87\`
    assert only that a trace contains each label, so these false boundary claims pass.
- Why this blocks adoption: the 20 snapshot schedules and 60 effect schedules are not exhaustive over
  the packet's required before/after commit, compensation, and event-publication boundaries. “No
  duplicate effect” is therefore partially tautological.
- Required correction: make each required boundary a real scheduler transition with pre-crash durable
  state, crash, restart and recovery. Add assertions about state/effect counts after each restart rather
  than accepting a trace label as coverage, then regenerate both evidence files.

### RW35-003 — W5 task/inbox retry classes contradict W3 writer evidence

- Lane/evidence: W5, \`P0.W5.COMMAND_CATALOG\` and \`P0.W5.EFFECT_RECOVERY_MATRIX\`
- Severity: critical
- Affected decisions: ADR-29, ADR-34
- Evidence:
  - \`scripts/hosted-web/phase-0/recovery-events/generate-evidence.mjs:28\` assigns
    \`idempotent_by_operation_id\` while simultaneously recording that current operation-ID proof is
    partial and not ADR-34-versioned.
  - \`scripts/hosted-web/phase-0/recovery-events/generate-evidence.mjs:86\` through \`:101\` applies that
    helper to inbox envelopes and task create/update/status/relationship file effects.
  - W3's \`docs/research/hosted-web/phase-0/state-writers/index.md:24\` through \`:30\` proves the task
    lock is process-local, classifies native task/config/inbox active writes as quiescent-only, and says
    transport delivery is not semantic success.
  - ADR-34 requires durable adapter/provider deduplication plus an equivalent lookup before
    \`idempotent_by_operation_id\`; a task ID or message ID surviving watcher echo does not prove that.
- Why this blocks adoption: W5 currently advertises automatic recovery for effects whose active writer
  and durable lookup are unproven. That is the exact cross-lane assumption reciprocal review must catch.
- Required correction: reconcile every task/inbox effect with W3's ADR-29 row. Use
  \`transactional_local\` only for a genuinely same-transaction internal effect; otherwise default to
  \`non_reconcilable/operator_required\` until a provider-mediated or cooperative adapter proves durable
  same-operation dedupe and lookup. Regenerate the matrix and ambiguous-effect list.

### RW35-004 — W5 fingerprint goldens do not cover the claimed version contract

- Lane/evidence: W5, \`P0.W5.FINGERPRINT_GOLDENS\`
- Severity: high
- Affected decisions: ADR-27, ADR-34
- Evidence:
  - \`scripts/hosted-web/phase-0/recovery-events/generate-evidence.mjs:264\` through \`:351\` includes
    field-order, changed-intent, schema-version, key-rotation, Unicode and integer cases.
  - It has no vector pair that changes \`fingerprintVersion\`, and no omitted-versus-materialized-default
    pair. The lone launch v1 case already includes the defaults.
  - \`test/architecture/hosted-web/phase-0/recovery-events/recovery-events.test.mjs:149\` through \`:160\`
    therefore cannot prove either requirement.
- Why this blocks adoption: the W5 acceptance text explicitly requires schema/fingerprint/key-version
  changes and retained comparison behavior; ADR-34 also requires omitted/default semantics.
- Required correction: add retained old fingerprint-version vectors, a new fingerprint-version vector,
  omitted-versus-materialized-default cases, arrays with non-empty ordered elements, and comparison
  outcomes across retained versions. Ensure the generator recomputes using the recorded historical
  encoder/key rather than the newest defaults.

## Cross-lane reconciliation

The W3 conclusion that active native task/config/inbox writes are uncoordinated and quiescent-only is
the controlling evidence until W2 supplies provider/version proof. W5 may not infer durable effect
idempotency from an application ID, desired bytes, watcher echo, or a transport-level delivery state.
W3's WAL Online Backup result supports W5's future single-SQLite-writer transaction seam only at
\`fixture_characterized\`; it does not prove the production async worker operation, final artifact ABI,
command keyring preservation, retention lease, or deployment-wide quiescence.

The shared \`EST-RECOVERY-STATE\` range must be deduplicated at controller integration. W3's 4.5k–7.25k
and W5's 4.5k–7.5k are overlapping decompositions, not additive estimates. Neither producer changed
the shared estimate ledger, production source, package metadata, lockfiles, or another lane.
